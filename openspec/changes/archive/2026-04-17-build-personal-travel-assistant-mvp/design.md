## Context

当前仓库是一个基于 Vite 的多页静态站点：首页在 `index.html`，攻略页位于 `guides/<slug>/index.html`，共享数据集中在 `src/data/trips.js`。现有能力更像“人工整理好的旅行作品集”，而不是“会和用户共同推敲方案的个人旅游助手”。

这次 change 的目标不是一步做成完整旅游平台，而是定义并落地第一阶段 MVP 的能力边界。MVP 需要同时兼顾两件事：一是让“做攻略”这件事明显更省力；二是保留项目当前已经具备的审美表达能力，继续生成适合手机分享的页面作品。

## Goals / Non-Goals

**Goals:**
- 支持围绕一个目的地进行多方案旅游路线讨论，而不是只产出单一答案。
- 支持把路线、交通方式、预算与酒店区域建议联动起来。
- 支持将确认后的方案生成精美可分享页面。
- 支持保留历史方案，并从历史页面与交互调整中总结用户偏好。
- 将第一阶段的产品边界固化为 OpenSpec 工件，便于后续编码与迭代。

**Non-Goals:**
- 不做订票、订酒店、支付等交易闭环。
- 不做自动代下单。
- 不做海外复杂行程。
- 不在本 change 中承诺上传打卡照片、自动生成 vlog、社交协作等扩展功能。

## Decisions

### 1. 以“能力拆分”组织规格，而不是直接按页面拆分
- 选择将 change 拆成 `collaborative-trip-planning`、`shareable-trip-pages`、`travel-preference-memory` 三个 capability。
- 原因：这样能把“讨论生成”“作品化展示”“历史学习”三条产品主线明确分开，避免后续实现时把交互层、展示层和偏好层耦合在一起。
- 备选方案：只写一个“大而全”的 spec。
- 不采用原因：会让需求边界和任务拆解都过于模糊，后续 apply 阶段难以分层推进。

### 2. 第一阶段优先做“决策支持”，不做“交易执行”
- 系统第一阶段聚焦于帮助用户判断值不值得去、选哪条路线、怎么去、住哪里。
- 原因：用户的首要痛点是“做攻略费劲”，不是“缺一个支付入口”。
- 备选方案：把订票和酒店预订也纳入 MVP。
- 不采用原因：会显著扩大系统复杂度和风险，还会偏离用户刚确认的非目标。

### 3. 偏好学习采用“显性历史 + 交互反馈”的组合模型
- 学习来源包括：既有页面中的结构化信息，以及用户在路线讨论中接受或拒绝的建议。
- 原因：仅靠历史页面会沦为内容复用，无法证明系统真的理解用户；仅靠即时反馈又无法积累长期风格。
- 备选方案：只使用历史页面做静态画像。
- 不采用原因：无法覆盖预算、家庭情况、夜间驾驶偏好、酒店档次风格等动态判断。

### 4. 决策边界前置进入产品设计
- 系统可以默认决定：交通路线、酒店区域、行程节奏。
- 系统必须先确认：出行人数、预算、出行风险。
- 系统不应过滤方案，而应排序+标注：被降权的方案仍然展示，但标注"根据您的偏好，此方案可能不太适合"，保留用户最终选择权。
- 原因：把授权边界提前写进规格，能减少未来交互设计与推荐逻辑的歧义；同时保持"多方案讨论"的透明度，避免系统过度自作主张导致用户错失更优选择。

### 5. 用“三层对象”承载规划、记忆与发布
- 第一阶段明确区分三类数据对象：`planDraft`、`preferenceProfile`、`trip`。
- `planDraft` 负责承载一次规划会话中的候选路线、交通对比、酒店区域建议以及讨论过程。
- `preferenceProfile` 负责跨多次规划沉淀用户偏好信号与证据来源。
- `trip` 负责承载最终已经确认并发布的分享页内容。
- 原因：当前仓库只有 `trip` 这一类“最终成品”数据，如果直接把候选方案和偏好学习硬塞进现有结构，后续会把讨论层、记忆层、展示层耦合在一起。
- 备选方案：继续只使用 `src/data/trips.js` 一类对象，同时把候选方案和偏好字段都塞进其中。
- 不采用原因：会导致中间态和终态混在一起，不利于生成链路、偏好提取和页面发布的分层演进。

### 6. 先跑通“规划中间层”，再增强信息智能
- MVP 先补一条从“输入条件”到“候选方案”再到“生成分享页”的最小链路。
- 第一阶段优先证明产品流程成立，不同时追求真实搜索、复杂偏好建模和自动发布全部到位。
- 原因：当前项目的终态页面渲染链路已经存在，真正缺的是候选方案这层中间态。
- 备选方案：直接把 AI 搜索、偏好学习、页面生成一次性做全。
- 不采用原因：在当前静态站基础上会迅速扩大复杂度，难以验证 MVP 到底是产品问题还是工程问题。

### 7. 先固定 MVP 技术占位，再进入页面与数据实现
- 第一阶段先明确三项实现前提：`信息源` 采用静态或示例数据、`偏好存储` 采用本地静态结构或 `localStorage` 占位、`/planner` 作为独立页面存在。
- 原因：如果把这些决策拖到实现中后期，前面的数据结构、状态流和页面交互都可能返工。
- 备选方案：先实现页面和数据模型，后续再决定信息源接入和存储落点。
- 不采用原因：这会让 `/planner`、`planDraft` 和 `preferenceProfile` 的代码边界不断变化，削弱 MVP 的验证效率。

## MVP Implementation Assumptions

这一 change 在进入实现阶段前，默认采用以下占位策略：

- `信息源`：MVP 首版接入轻量级实时 API 驱动核心能力，具体包括：
  - **高德地图 POI API**：获取景点、餐厅、酒店等 POI 信息及评分（免费额度 30 万次/日）
  - **高德路径规划 API**：计算驾车/公交路线的实时距离与时长（免费额度 30 万次/日）
  - **和风天气 API**：提供未来 7 天天气预报与灾害预警（免费额度 1000 次/日）
  - **节假日 API**：查询法定节假日与调休安排（完全免费，数据源：https://timor.tech/api/holiday）
  - **静态旺季数据**：手动维护热门目的地的旺季/淡季时间表，作为 API 的补充
  - **降级策略**：API 调用失败时回退到静态示例数据，并明确标注”当前使用历史数据”；关键信息缺失时提示用户自行核实
  - **时效性声明**：在规划页面底部标注信息来源与时效性，提示用户”景区开放状态建议出行前核实官方信息”

- `偏好存储`：MVP 首版使用浏览器侧 `localStorage` 作为 `preferenceProfile` 的最小存储落点，不引入后端持久化。偏好模型包含以下机制：
  - **冷启动处理**：新用户首次使用时，通过规划表单中的”出行人数、预算、兴趣标签”快速建立初始画像
  - **置信度模型**：每条偏好记录包含置信度分数（0-1），基于证据数量与一致性动态调整
  - **时间衰减**：预算类偏好设置 6 个月衰减期，长期偏好（如”不喜欢爬山”）不衰减
  - **噪声过滤**：单次”拒绝建议”不立即更新偏好，需 3 次以上一致信号才写入 profile
  - **冲突处理**：当历史偏好与本次明确要求冲突时，优先采用本次输入，并在界面标注”本次偏好与历史不同”

- `规划页面`：`/planner` 作为独立页面存在，负责承载输入、候选方案和再次调整，不与首页混合成单页工作台。

这些假设的目的不是长期定型，而是保证第一阶段能先验证”做攻略更省力”的产品闭环。二期若需升级信息源（如接入携程价格 API）或引入轻量后端存储，应在不破坏 `planDraft -> trip` 主链路的前提下替换占位实现。

## Data Model Boundaries

### `planDraft`

用于表示一次旅游规划会话中的中间态数据，是“共同探讨”能力的主要载体。

建议至少包含：
- 基础输入：目的地、出发地、天数、人数、预算、日期
- 偏好约束：交通偏好、节奏、兴趣、酒店风格、明确回避项
- 候选方案列表：A/B/C 路线、适配理由、交通对比、酒店区域建议、粗略行程
- 讨论日志：用户接受了什么、拒绝了什么、哪些约束被新增或收紧

### `preferenceProfile`

用于表示跨多次规划累积出来的用户偏好记忆，是”学到你了”的最小载体。

建议至少包含：
- 稳定偏好：常用天数、交通倾向、回避规则、酒店风格、内容偏好
- 推断偏好：预算带、家庭情况、行程节奏等
- 证据来源：来自历史 `trip` 的显性信号，以及来自讨论过程的行为反馈
- 置信度与时间衰减：每条偏好包含置信度分数（0-1）与最后更新时间，预算类偏好 6 个月后衰减，长期偏好不衰减

用于表示最终已经确认并发布的旅行页面内容，是当前仓库中已经存在的终态对象。

建议在现有结构基础上补充：
- 来源字段：`sourcePlanId`、`sourceOptionId`
- 决策摘要：选中了哪种交通、否决了什么、推荐了哪个酒店区域
- 使用过的偏好信号
- 数据来源标注：标记哪些信息来自实时 API，哪些来自静态数据

### 数据流关系

```text
用户输入
  -> planDraft
  -> 讨论与调整
  -> 选中一个 option
  -> 映射为 trip
  -> 发布为分享页

历史 trip + 讨论反馈
  -> preferenceProfile
  -> 影响下一次 planDraft
```

## MVP Page Flow

第一阶段建议引入一条最小新链路，而不是重写首页和详情页：

```text
首页
├─ 已发布旅程合集
└─ 开始规划入口

/planner
├─ 输入目的地、人数、预算、日期、偏好约束
└─ 进入候选方案结果

/planner/result
├─ 展示 A/B/C 候选路线
├─ 展示交通对比
├─ 展示酒店区域建议
└─ 支持再次调整偏好

/guides/<slug>/
└─ 复用现有视觉系统展示最终分享页
```

这样可以最大化复用现有 `home.js`、`guide.js` 和既有视觉样式，同时把新增复杂度集中在“规划层”。

## UI Contract: `/planner`

`/planner` 是第一阶段新增的规划入口页，职责是采集必要约束、展示候选方案，并把“已选方案”推进到最终分享页生成链路。

### 1. 表单字段

建议最小表单分为“必须确认字段”和“可调整偏好字段”两组。

必须确认字段：
- `destination`：目的地，文本输入，必填
- `origin`：出发地，文本输入，必填
- `days`：出行天数，数字或选项，必填
- `partySize`：出行人数，数字输入，必填
- `budgetRange`：预算区间，区间输入或选项，必填
- `travelDate`：出行日期或时间窗，必填
- `riskPreference`：出行风险偏好，单选，必填

可调整偏好字段：
- `transportBias`：交通偏好，多选，例如自驾、高铁、飞机
- `pace`：行程节奏，单选，例如松弛、适中、紧凑
- `interests`：兴趣偏好，多选，例如风景、人文、夜景、美食
- `hotelStyle`：酒店风格，多选，例如方便、省心、有氛围、中高档
- `avoidRules`：明确不要的体验，多选或自由输入，例如夜间开车、过密赶路、过度商业化

### 2. 表单提交结果

- 当必须确认字段缺失时，页面不得进入最终推荐态，必须提示用户补齐。
- 当用户提交表单后，系统进入“候选方案结果区”，展示 A/B/C 方案，而不是直接生成最终分享页。
- 当用户再次修改偏好字段后，系统应重新生成或刷新候选方案结果。

### 3. 候选方案卡片结构

每张候选方案卡片至少应包含以下固定区域：
- `title`：方案标题
- `summary`：一句话摘要
- `fitReasons`：为什么适合当前用户的 2-3 条原因
- `routeTags`：方案标签，例如风景、经典、高铁友好
- `transportRecommendation`：交通推荐摘要
- `stayAreaRecommendation`：酒店区域建议摘要
- `paceLabel`：行程节奏说明
- `itineraryOutline`：按天或按阶段的粗略行程
- `actions`：至少包含“选择这个方案”和“继续调整偏好”两个动作

### 4. 候选方案卡片行为

- 点击“选择这个方案”后，系统将当前方案标记为 `selectedOptionId`，并允许进入最终分享页生成流程。
- 点击“继续调整偏好”后，页面应回到表单或保留表单在同页可编辑状态，不应丢失已填写信息。
- 若系统根据 `preferenceProfile` 使用了历史偏好，卡片中应展示简短推荐理由，例如“基于你偏好高铁短途和避免夜间驾驶”。

### 5. 页面状态

`/planner` 至少需要有以下几种状态：
- `idle`：初始输入态，尚未生成候选方案
- `loading`：正在生成候选方案
- `results`：已生成候选方案
- `selected`：用户已选中一个方案，准备进入最终分享页生成
- `error`：信息不足或生成失败，需要提示修正

### 6. 与数据模型的对应关系

- 表单字段主要映射到 `planDraft.destination`、`planDraft.travelerContext`、`planDraft.preferences`
- 候选方案卡片主要映射到 `planDraft.options[*]`
- 选中动作主要更新 `planDraft.selectedOptionId`
- 历史偏好提示主要来自 `preferenceProfile`

## Mobile UX Requirements

面向移动端用户，MVP 需满足“低学习成本、低误触成本、短路径完成发布”的体验目标。

### 1. 流程与信息架构

- 采用分步流程：`必填信息 -> 偏好微调 -> 候选方案 -> 发布确认 -> 发布完成`，避免单屏信息过载。
- 候选方案阶段必须保留“重新调整再生成”路径，不得强制线性提交。
- 发布动作必须包含确认层，减少误触直接跳转。

### 2. 操作效率与触达体验

- 底部使用 sticky 主操作区，核心 CTA 在单手拇指热区内可达。
- 主按钮高度建议不低于 `44px`，同屏仅保留一个主 CTA，其他动作用次级按钮呈现。
- 首次填写前提供快捷模板，降低表单输入成本。
- 检测到草稿时提供“继续草稿 / 清空重来”双入口，支持中断后继续。

### 3. 决策透明度与内容表达

- 选择方案后需展示“与次优方案差异提示”，帮助用户理解取舍原因。
- 文案需统一为“用户任务语言”，避免技术术语与系统内部概念堆叠。
- 在候选方案与发布结果区明确边界：本系统提供决策支持，不进入交易与代下单。

### 4. 发布后闭环

- 发布完成后展示独立成功态，至少包含“查看分享页”“返回首页”两个后续动作。
- 首页列表需即时反映新发布条目，确保用户对“已发布成功”形成可见反馈。

### 5. 可测试验收项

- 自动化测试需覆盖：快捷模板、分步切换、候选方案生成、差异提示、发布确认、发布成功后跳转。
- 自动化测试需输出页面截图，覆盖首页、候选方案页、分享页三个关键节点。

## MVP 补充优化记录（2026-04-17）

基于移动端实测与连续回归，本次补充优化纳入需求基线，作为 MVP 第一版交付范围的一部分：

- 生成阶段交互：点击“生成方案”后必须先渲染“查询中”状态，再发起实时请求，避免页面短暂无反馈。
- 交通决策闭环：候选方案需支持“按方案默认推荐 + 用户手动覆盖交通”；发布结果以用户最终选中交通为准。
- 交通信息表达：自驾展示统一为“路线 + 实时里程 + 高速费”，去除重复描述；高铁/飞机展示按“人均费用”口径表达。
- 实时信息可信度：天气展示按“出行起始日 + 出行天数”窗口输出逐日明细，并带天气图标；节假日展示按真实出发日判断，不得出现节日误标。
- 来源可见性：高德实时数据与 AI 生成内容需有清晰来源标识，避免用户误解数据来源。
- 移动端视觉修订：日期选择控件提升可读与点按体验；交通选中高亮采用克制样式，确保“可见但不抢主流程”。
- 文案可理解性：对内部标签和技术术语做用户语言映射，推荐理由和路线描述需可读、可执行。

## Risks / Trade-offs

- [风险] 当前仓库还是静态站结构，直接承载”讨论式规划”会让信息流与页面流混杂
  → Mitigation：在规格与任务层先明确数据模型与页面职责，再进入实现。
- [风险] “偏好学习”概念容易被做成旧内容拼装
  → Mitigation：在 specs 中要求系统必须体现预算、家庭、兴趣、驾驶与酒店偏好等明确维度，并引入置信度模型与时间衰减机制。
- [风险] 用户对客观信息的时效性可能有较高期待
  → Mitigation：接入高德地图、和风天气等实时 API，并在页面底部明确标注信息来源与时效性声明；API 失败时启用降级策略并标注”当前使用历史数据”。
- [风险] 免费 API 额度可能不足以支撑用户增长
  → Mitigation：高德地图 30 万次/日额度可支撑约 1 万日活用户（假设每用户 30 次调用）；和风天气 1000 次/日可支撑约 200 日活；若超出额度，优先降级天气查询，保留核心 POI 与路径规划能力。
- [风险] 偏好学习的冷启动与噪声过滤可能影响推荐准确性
  → Mitigation：新用户通过表单快速建立初始画像；单次交互不立即更新偏好，需 3 次以上一致信号；置信度低的偏好在推荐时降权而非过滤。

## Migration Plan

- 第一步：注册并配置外部 API（高德地图、和风天气），验证接口可用性与免费额度，编写 API 调用封装层与降级逻辑。
- 第二步：整理现有静态数据与页面模板，使其能承载”方案草案”和”历史方案”；定义 `planDraft`、`preferenceProfile`、`trip` 三层数据结构。
- 第三步：先引入 `/planner` 规划中间层，接入实时 API 生成候选方案，验证”输入条件 → 候选方案”的核心链路。
- 第四步：打通 `planDraft -> trip` 的分享页生成链路，确保用户可以将确认的方案发布为精美页面。
- 第五步：在不破坏现有攻略展示页的前提下，把新生成旅程接回首页合集，并补偏好记忆 MVP（包含冷启动、置信度、时间衰减机制）。
- 回滚策略：若新能力尚未稳定或 API 调用异常，可继续保留现有静态攻略页作为兜底展示路径；若 API 额度耗尽，自动降级到静态示例数据并标注。

- 偏好学习中的”3 次一致信号”阈值是否需要根据用户活跃度动态调整（高频用户可能需要更高阈值）。
